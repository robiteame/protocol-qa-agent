# 01 — 总体架构

## 1. 系统定位

输入：用户上传的 Office 文档（.docx / .xlsx / .pptx，可扩展 PDF）——通常是通讯协议规范书、寄存器表、报文格式说明。
输出：针对用户提问，从文档中检索相关片段，由 LLM 生成**带出处引用**的回答。

通讯协议文档的特点决定了技术方案：

| 文档特点 | 带来的问题 | 应对方案 |
|----------|-----------|---------|
| 大量表格（寄存器表、字段定义、报文格式） | 普通文本切块会撕碎表格，向量检索召回质量差 | 表格整体作为一个块；表格转 Markdown 保留结构 |
| 层级结构强（章节 → 小节 → 字段） | 片段脱离上下文后含义丢失（"地址 0x03"属于哪个功能码？） | 切块时附带"章节路径"元数据；PageIndex 树保留全局结构 |
| 术语缩写多（CRC、PDU、ADU…） | 用户口语化提问与文档措辞不匹配 | 查询改写（Query Rewrite）+ 混合检索（向量+关键词） |
| 跨章节关联（第3章定义帧格式，第7章定义错误码） | 单次检索不够，需要多跳 | Agent 多轮检索循环 + PageIndex 树状推理 |

## 3. 技术选型与理由

| 层 | 选型 | 理由（以及为什么不选别的） |
|----|------|--------------------------|
| Web 框架 | FastAPI | 自带 Swagger 文档、Pydantic 校验、async 支持；比 Flask 更贴近现代工程实践 |
| 文档解析 | python-docx / openpyxl / python-pptx | 各格式的事实标准库；解析属于"脏活"，用库，不影响学习核心 |
| 向量库 | ChromaDB（本地嵌入模式） | 零部署、pip 装完即用、支持元数据过滤；学习阶段不需要 Milvus/Qdrant 的运维成本 |
| 关键词检索 | 自实现 BM25（或 rank-bm25 库） | 协议文档术语精确匹配很重要；自己实现一遍 BM25 是极好的 Python 练习 |
| 结构化存储 | SQLite + 标准库 sqlite3 | 零部署；练习 SQL 与 Python DB-API；后期可换 SQLAlchemy |
| 模型接入 | 自写适配层（抽象基类 + 两个实现） | 练习 Python 的 ABC/Protocol、依赖注入思想；OpenAI 兼容 API 与 Ollama 都只是 HTTP 调用 |
| HTTP 客户端 | httpx | 同时支持同步/异步，与 FastAPI 风格一致 |
| Agent 框架 | **不用框架，手写循环** | 核心学习目标；Agent 本质 = while 循环 + LLM 决策 + 工具调用 |
| 配置管理 | pydantic-settings + .env | 类型安全的配置读取，业界通行做法 |

> **手写边界原则**：凡是"学习价值高"的部分（Agent 循环、RAG 编排、BM25、PageIndex 构建、融合排序）一律手写；凡是"格式解析类脏活"（docx 解析、向量存储底层）用库。

## 4. 两条核心数据流

### 4.1 入库流（Ingestion，写路径）

```
用户上传 file.docx
  │
  ▼
[API层] 校验格式/大小 → 计算文件哈希（去重）
  │
  ▼
[存储] 原始文件落盘 data/raw/{doc_id}/file.docx
       SQLite 写入 documents 表，状态 = parsing
  │
  ▼
[解析器] docx → 中间表示 IR（带层级的元素列表：标题/段落/表格）
  │            （这是关键设计：所有格式都归一到同一个 IR，下游只认 IR）
  ▼
[切块器] IR → chunks（结构感知：按章节切、表格不拆、带元数据）
       SQLite 写入 chunks 表
  │
  ├──────────────────────┬─────────────────────────┐
  ▼                      ▼                         ▼
[向量化]              [BM25 索引]              [PageIndex 构建]
chunk文本→embedding   分词→倒排/词频统计        LLM 为文档生成树状目录
存入 ChromaDB         （内存构建+落盘）         （节点=章节，含摘要）
  │                      │                    存为 data/pageindex/{doc_id}.json
  └──────────────────────┴─────────────────────────┘
  │
  ▼
SQLite 更新文档状态 = ready（任一步失败则 = failed + 错误信息）
```

设计要点：
- **doc_id 用 UUID**，文件哈希用于去重提示（同一文件重复上传时提示而非报错）。
- 入库是耗时操作（Embedding、LLM 建树都要调模型），API 必须**立即返回**，入库放后台任务（FastAPI `BackgroundTasks` 即可，学习阶段不必上 Celery）。
- 文档状态机：`uploaded → parsing → chunking → embedding → indexing → ready / failed`，前端轮询状态。

### 4.2 问答流（Query，读路径）

```
用户提问："Modbus 读保持寄存器的功能码是多少？异常响应怎么判断？"
  │
  ▼
[查询理解] LLM 改写：拆出 2 个子问题 + 补全术语
  │        ("读保持寄存器 功能码 0x03"、"异常响应 功能码最高位")
  ▼
[检索调度] 对每个子查询并行执行：
  │   ├─ 向量检索（ChromaDB top-k）
  │   ├─ BM25 检索（关键词 top-k）
  │   └─ PageIndex 检索（LLM 在树上导航：根→相关章节→定位节点→取对应chunks）
  ▼
[融合重排] RRF（倒数排名融合）合并三路结果 → 去重 → 取 top-n
  │
  ▼
[充分性判断] LLM 判断：现有片段够回答吗？
  │     ├─ 不够 且 循环次数 < 上限 → 生成新查询，回到[检索调度]   ← Agent 循环！
  │     └─ 够了 ↓
  ▼
[生成回答] System Prompt + 检索片段（带编号） + 对话历史 + 用户问题
  │        要求 LLM：只依据片段作答、标注引用 [1][2]、不知道就说不知道
  ▼
返回答案 + 引用来源（文档名/章节路径/原文片段）
```

设计要点：
- Agent 循环上限设 3 轮，防止失控烧 token。
- 每一步都记录到 `trace`（中间过程日志），调试与展示"思考过程"都靠它。
- 对话历史存 SQLite，多轮提问时把近几轮 Q/A 摘要进上下文。

## 5. 关键设计决策记录（ADR 摘要）

1. **为什么双引擎而非只用向量 RAG？**
   向量检索擅长语义模糊匹配，但对"这个文档第几章讲了什么""帧结构整体是怎样的"这类**全局性/结构性问题**无能为力——因为它只见树木不见森林。PageIndex 保留文档树状结构，让 LLM 像人一样"翻目录找章节"，正好补这个短板。

2. **为什么先定义中间表示（IR）？**
   三种 Office 格式解析结果各异，如果切块器直接依赖 python-docx 的对象，将来加 PDF 就要重写切块器。IR 是典型的"防腐层"设计，也是接口设计的练习点。

3. **为什么 PageIndex 树用 LLM 构建而不是纯规则？**
   规则（按标题样式分层）作为骨架，LLM 负责生成每个节点的**摘要**——摘要质量决定了树上导航的准确率。骨架规则化、内容智能化，成本与效果的平衡。

4. **为什么不用 LangChain？**
   学习目标是理解 Agent 与 RAG 的本质机制。框架封装会把"检索→判断→再检索"这个最有价值的循环藏起来。等你手写过一遍，再看框架会一目了然。
