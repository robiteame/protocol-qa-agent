# 02 — 检索技术专章：传统 RAG 与 PageIndex

> 本章是检索层的"原理 + 设计"说明。实现检索相关模块（retrieval/ 目录）前必读。

## 1. 传统 RAG：向量检索 + 关键词检索

### 1.1 向量检索（语义检索）原理

核心思想：把文本映射到高维向量空间，**语义相近的文本，向量距离近**。

```
"读保持寄存器用什么功能码"  ──Embedding模型──►  [0.12, -0.83, 0.45, ...]（如1024维）
"功能码 0x03：Read Holding Registers" ──►       [0.15, -0.79, 0.41, ...]
                                                  ↑ 两个向量余弦相似度高 → 召回
```

入库时：每个 chunk 文本 → Embedding → 存入 ChromaDB。
查询时：问题 → 同一个 Embedding 模型 → 在库里找最近的 top-k 个向量 → 返回对应 chunk。

**你需要理解但不需要实现的部分**：相似度计算（余弦相似度）、近邻搜索——ChromaDB 替你做。
**你需要实现的部分**：批量向量化的调度（注意 API 有单次条数限制，要分批）、入库与查询的封装。

关键认知：
- 查询和文档**必须用同一个 Embedding 模型**。换模型 = 整库重建。把模型名记入文档元数据，加载时校验。
- 向量检索的强项是"语义模糊匹配"（用户说"读寄存器"，文档写"Read Holding Registers"也能召回）。
- 弱项：**精确术语/数字匹配不可靠**。问"0x10 功能码"，向量可能召回 0x03、0x06 的段落——数字在语义空间里区分度低。这就是要加 BM25 的原因。

### 1.2 BM25 关键词检索（建议手写，最佳 Python 练习）

BM25 是经典的词频统计排序算法，给定查询词，对每个文档（这里是 chunk）打分：

```
score(Q, D) = Σ_{词t∈Q} IDF(t) × ( f(t,D) × (k1+1) ) / ( f(t,D) + k1 × (1 - b + b × |D|/avgdl) )
```

各符号直觉含义（不必死记，理解即可）：
- `f(t,D)`：词 t 在 chunk D 里出现几次——出现越多越相关，但有饱和（k1 控制饱和速度，常取 1.5）
- `IDF(t)`：词 t 在多少 chunk 里出现过——越罕见的词信息量越大（"CRC16" 比 "数据" 值钱）
- `|D|/avgdl`：chunk 长度归一化——长 chunk 天然包含更多词，要惩罚（b 控制惩罚力度，常取 0.75）

**实现路线**（详见 05 阶段 4）：
1. 分词：中文用 `jieba`，英文/十六进制术语按空白与标点切分，统一转小写。注意"0x03"、"CRC-16"这类 token 要保留完整，不要被标点切碎——自定义一个 token 保护规则。
2. 建索引：遍历所有 chunk，统计每词的文档频率（`collections.Counter` 的用武之地）+ 每 chunk 的词频表。
3. 查询打分：对查询分词，按公式对候选 chunk 打分，排序取 top-k。
4. 持久化：索引结构用 `json` 或 `pickle` 落盘，文档增删后重建（学习阶段全量重建即可，不做增量）。

> 偷懒选项：`rank-bm25` 库三行搞定。但这是整个项目里**性价比最高的手写练习**——涉及分词、字典操作、数学公式转代码、排序，强烈建议自己写，再用库的结果对拍验证。

### 1.3 为什么要混合检索

| 查询类型 | 向量检索 | BM25 |
|----------|---------|------|
| "怎么判断从机有没有正常响应"（口语化、语义）| ✅ 强 | ❌ 词对不上 |
| "0x83 异常码含义"（精确术语）| ⚠️ 不可靠 | ✅ 强 |
| "CRC 校验放在帧的什么位置"（混合）| ✅ | ✅ |

协议文档充满精确术语（功能码、寄存器地址、字段名），两路检索**互为保险**。

## 2. PageIndex：树状推理检索

### 2.1 解决什么问题

传统 RAG 的根本局限：**把文档打碎成无结构的块**。以下问题它答不好：

- "这份协议文档总共定义了哪几类报文？"（需要全局视野，没有任何单个 chunk 包含答案）
- "第 5 章和第 7 章关于超时的规定有什么区别？"（需要按结构定位，而非按语义模糊匹配）
- "异常响应的完整处理流程是什么？"（答案横跨连续多节，碎块召回会丢上下文）

PageIndex 的思路（源自 [PageIndex 论文/开源项目](https://github.com/VectifyAI/PageIndex) 的理念）：**不切碎文档，而是为文档建一棵"目录树"，让 LLM 像人一样翻目录定位章节**。这被称为 *reasoning-based retrieval*（推理式检索）——靠 LLM 的推理而非向量相似度来找信息。

### 2.2 树的结构设计

每个文档入库时生成一棵树，存为 `data/pageindex/{doc_id}.json`：

```json
{
  "doc_id": "…",
  "title": "Modbus应用协议规范V1.1b",
  "root": {
    "node_id": "0000",
    "title": "(root)",
    "summary": "本文档定义了Modbus应用层协议……（LLM生成的全文摘要）",
    "children": [
      {
        "node_id": "0001",
        "title": "3 帧格式定义",
        "summary": "本章定义ADU/PDU结构，包含地址域、功能码……",
        "chunk_ids": ["c-101", "c-102"],
        "children": [
          {
            "node_id": "0002",
            "title": "3.1 RTU模式帧结构",
            "summary": "……",
            "chunk_ids": ["c-103", "c-104", "c-105"],
            "children": []
          }
        ]
      }
    ]
  }
}
```

设计要点：
- **节点 = 文档章节**。骨架来自解析阶段提取的标题层级（docx 的 Heading 样式 / pptx 的页标题 / xlsx 的 sheet 名）——规则化提取，不靠 LLM。
- **summary 由 LLM 生成**，这是树的灵魂：导航时 LLM 只看 title + summary 就要判断"答案可能在这个分支吗"。摘要要写明"本节包含哪些信息点"，而不是泛泛概括。
- **chunk_ids 把树叶挂回 chunk 表**：树负责"定位到章节"，最终送给生成模型的还是该章节下的 chunks（与传统 RAG 共用同一份 chunk 数据，一份切块两套索引——这是本架构的关键复用点）。
- 节点过大时（如一章 50 个 chunk）摘要生成采用"先分段摘要、再汇总摘要"的 map-reduce 两级策略。

### 2.3 树上导航检索流程

```
查询："异常响应的帧格式和正常响应有什么区别？"
  │
  ▼
第1步：把 根节点的所有子节点(title+summary) 列给 LLM
        Prompt 要求其输出：相关的 node_id 列表（可多选）+ 理由
        → LLM 选中 ["3 帧格式定义", "7 异常响应"]
  │
  ▼
第2步：对每个选中节点，若有子节点 → 递归下钻（重复第1步逻辑）
        若是叶子/LLM判断"本节点整体相关" → 停止，收集该节点 chunk_ids
  │
  ▼
第3步：返回收集到的 chunks（附带"树路径"作为出处：第3章>3.1节）
```

实现要点：
- 这是一个**递归/迭代下钻**过程，每层一次 LLM 调用。限制最大深度（如 3 层）与最大选中节点数（如每层 ≤3），控制成本。
- LLM 输出要求 JSON 格式（`{"selected": ["0001","0007"], "reason": "..."}`），你需要写**健壮的 JSON 解析**：LLM 可能输出多余文字，要先用正则/字符串处理提取 JSON 部分，解析失败要重试或降级（降级策略：返回空，让向量检索兜底）。这是 LLM 工程的必修课。
- 允许 LLM 回答"没有相关章节"——此时 PageIndex 路返回空结果，靠另两路兜底。

### 2.4 两引擎能力对比（何时谁强）

| 维度 | 传统 RAG（向量+BM25） | PageIndex |
|------|---------------------|-----------|
| 细节事实问题（"0x03 是什么"）| ✅ 快、准 | ⚠️ 杀鸡用牛刀 |
| 全局/结构问题（"文档讲了哪几块"）| ❌ | ✅ |
| 跨章节比较 | ⚠️ 碰运气 | ✅ 可同时下钻多分支 |
| 延迟/成本 | 低（一次 embedding + 本地查询）| 高（每层一次 LLM 调用）|
| 可解释性 | 弱（相似度分数说不清）| 强（有导航路径和理由）|

结论：**两路并行执行，融合取长**。简单问题向量路先出结果，结构问题 PageIndex 补盲区。

## 3. 多路融合：RRF（Reciprocal Rank Fusion）

三路检索各自返回一个**有序列表**，分数体系互不可比（余弦相似度 vs BM25 分 vs 树路径无分数）。RRF 只看**排名**，天然适合异构融合：

```
RRF_score(chunk) = Σ_{每路检索r} 1 / (k + rank_r(chunk))     k 常取 60
```

示例：chunk A 在向量路排第 1、BM25 路排第 3、PageIndex 路未出现：

```
score(A) = 1/(60+1) + 1/(60+3) + 0 ≈ 0.0164 + 0.0159 = 0.0323
```

实现要点：
- 输入：`dict[来源名, list[chunk_id 有序列表]]`；输出：按 RRF 分排序的去重列表。十几行代码，但要想清楚"chunk 缺席某路"的处理（贡献 0）。
- PageIndex 路返回的 chunks 没有内部排序，可按"树路径深度优先顺序"给定名次。
- 可给 PageIndex 路加权（如 ×1.2），鼓励结构化证据——做成可配置参数，方便后期调参实验。
- 融合后取 top-n（建议 6~8 个 chunk）进入生成阶段，超过的截断——上下文不是越多越好，无关片段会稀释 LLM 注意力。

## 4. 各环节 Prompt 设计要点

> 只给结构框架与要素清单，措辞自己写、自己迭代——调 Prompt 是 Agent 开发的核心技能，效果不好时回到这里逐条检查。

### 4.1 查询改写（Query Rewrite）
- 输入：用户原始问题 + 近几轮对话摘要
- 要求 LLM 输出：1~3 个检索友好的子查询（JSON 数组）
- 要素：补全指代（"那它的响应呢" → 补全"它"指什么）；展开/补充术语（口语 → 文档措辞）；复合问题拆分
- 约束：禁止编造原问题没有的概念；输出严格 JSON

### 4.2 树导航（PageIndex Navigation）
- 输入：当前层各节点的 node_id + title + summary，以及查询
- 要求输出：选中的 node_id 数组 + 一句话理由；明确允许"空选"
- 关键约束：只能从给出的 node_id 中选，最多选 N 个

### 4.3 充分性判断（Sufficiency Check）
- 输入：查询 + 当前已收集片段（编号列表）
- 要求输出 JSON：`{"sufficient": bool, "missing": "缺什么信息", "next_queries": [...]}`
- 这是 Agent 循环的"决策大脑"。注意提示它"宁可判充分也不要无限要更多"——LLM 天然倾向说"还不够"

### 4.4 最终生成（Answer Generation）
- System Prompt 要素：角色（通讯协议技术助手）；**只依据提供的片段作答**；每个论断标注引用编号 [1][2]；片段不含答案时明确说"文档中未找到"，禁止用自身知识补充协议细节（协议数字搞错比不答更糟）；表格类信息尽量用表格还原
- User 部分结构：`[片段1] (出处: 文档名>章节路径)\n内容…` × n + 对话历史摘要 + 当前问题

## 5. 评测意识（贯穿开发）

从阶段 4 起就维护一个 `eval/questions.jsonl`：20~30 个针对你测试文档的问题 + 期望答案要点 + 期望出处章节。每次改动检索逻辑后跑一遍，肉眼对比（后期可让 LLM 当裁判打分）。**没有评测集，调参就是盲人摸象。**

问题类型要覆盖：细节事实型 / 表格查询型 / 全局结构型 / 跨章节比较型 / 文档中不存在答案型（测试"不知道就说不知道"）。
